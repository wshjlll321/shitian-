# aerospace-product-storytelling Skill

## 负责什么

负责无人机产品能力叙事、航空工业可信度、工程感表达和复杂场景作业价值表达。它决定产品页、首页产品矩阵、参数模块、技术能力模块和案例关联方式。

本 Skill 的目标是让产品看起来像高端航空装备，而不是普通植保无人机商品。

## 必须坚持什么

- 产品表达必须从任务出发，再进入参数。
- 每个型号都必须有清晰角色：T280 是重载复杂任务平台，S270 是长航时农林作业平台，H15 是轻型巡查平台，H60 是高效电动植保平台，HIZONE 是低空作业数字中枢。
- 产品参数必须转译成场景价值。
- 航空工业可信度必须通过结构、动力、载荷、航时、控制半径、升限、环境适应性和作业案例建立。
- 技术表达必须准确，不得制造虚假能力。
- 产品页必须有“定位、关键能力、任务场景、核心参数、案例验证、咨询入口”的结构。
- 必须突出燃油无人直升机相比普通多旋翼在载荷、航时、作业半径和复杂地形中的优势。
- 必须让 HIZONE 与无人机作业闭环发生关系，避免孤立的软件展示。

## 禁止做什么

- 禁止把产品页做成电商详情页。
- 禁止只堆参数，不解释参数对任务的意义。
- 禁止用“黑科技”“颠覆”“遥遥领先”等空泛词。
- 禁止把所有产品平均展示，导致核心产品失焦。
- 禁止夸大飞行安全、自动驾驶、AI 决策等能力。
- 禁止用战斗机、军工、科幻武器等不相关视觉替代民用航空装备气质。
- 禁止让 Three.js 模型炫技压过产品本身。
- 禁止隐藏关键规格或把参数做得不可读。

## 如何检查质量

- 用户是否能快速理解每个产品适合什么任务。
- 关键参数是否被转化为真实作业价值。
- 产品能力是否能被案例证明。
- 技术语言是否准确、克制、专业。
- 参数表是否统一单位、格式和命名。
- 产品矩阵是否有主次，T280 / S270 是否建立足够强的品牌记忆。
- HIZONE 是否被表达为作业系统的一部分，而不是孤立页面。

## 给后续代码生成的具体约束

- 产品数据必须结构化，包含型号、定位、载荷、航时、速度、半径、升限、尺寸、作业场景和关联案例。
- 产品卡片必须显示任务定位，不得只显示名称和图片。
- 产品详情页必须提供参数表和关键能力模块，二者不能互相替代。
- 参数表必须支持移动端阅读，不能用超宽表格直接溢出。
- 每个产品至少关联一个真实场景或案例；无案例的产品必须标记为待补充。
- 首页产品区必须突出核心机型，不得平均分配视觉权重。
- 与飞行能力相关的文案必须能追溯到旧官网参数或用户确认资料。
