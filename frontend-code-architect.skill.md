# frontend-code-architect Skill

## 负责什么

负责新官网的前端代码架构，包括 Next.js、Tailwind、Framer Motion、Three.js、内容数据结构、组件边界、性能策略、可维护性、响应式实现和后续扩展方式。

本 Skill 只在正式编码阶段使用，但它会提前约束内容、设计系统和交互方案，避免后续代码混乱。

## 必须坚持什么

- 代码必须围绕内容资产、设计系统和品牌叙事组织，而不是先写页面再塞内容。
- Next.js 页面结构必须清晰，适合首页、产品、场景、案例、关于、联系等官网信息架构。
- Tailwind 必须基于设计令牌使用，不能到处写临时颜色和尺寸。
- Framer Motion 必须服务滚动叙事和状态过渡，不做廉价动效。
- Three.js 必须懒加载、可降级、可维护。
- 产品、案例、场景、联系方式必须数据化管理。
- 组件必须按业务含义命名，不得只有视觉命名。
- 代码必须兼顾桌面端高端视觉和移动端可读性。
- 性能、SEO、可访问性和图片优化必须作为官网基础能力。

## 禁止做什么

- 禁止在一个页面文件里堆满所有内容和样式。
- 禁止硬编码重复产品参数、联系方式和案例数据。
- 禁止用随机 class 拼出不可维护的视觉系统。
- 禁止滥用 client component。
- 禁止全站 heavy animation。
- 禁止 Three.js 阻塞首屏。
- 禁止没有 fallback 的视频、图片和 WebGL。
- 禁止为了动画牺牲可读性、SEO 和性能。
- 禁止使用未统一的按钮、卡片、表单和参数表实现。

## 如何检查质量

- 数据、组件、页面、样式和动效是否边界清晰。
- 修改一个产品参数是否只需要改一个数据源。
- 联系方式是否只需要改一个配置。
- 页面是否具备合理 metadata 和语义结构。
- 首屏是否避免加载非必要 3D 和大视频。
- 移动端布局是否独立审查过。
- 组件是否能支持后续增加产品、案例和场景。
- 动效是否尊重 reduced motion。
- 构建、类型检查和核心页面渲染是否通过。

## 给后续代码生成的具体约束

- 编码前必须先建立：内容迁移表、Rules、Design System、页面信息架构和组件清单。
- 必须使用结构化数据文件管理 products、scenarios、cases、contacts、navigation 和 media。
- 必须建立设计令牌文件或 Tailwind theme 扩展，不得散写主色。
- 必须建立基础组件：Button、Section、ProductSpecTable、MetricStrip、MediaBlock、CaseCard、ScenarioPanel、InquiryCTA、ContactForm。
- Three.js 相关代码必须与普通页面组件分离，并动态导入。
- 图片必须使用优化组件或明确尺寸，避免布局跳动。
- 页面模块必须支持响应式约束，不能只适配桌面端。
- 所有动画必须有 reduced motion 分支。
- 提交前必须至少运行构建或类型检查；如无法运行，必须说明原因。
